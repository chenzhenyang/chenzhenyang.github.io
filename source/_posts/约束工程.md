# 约束工程

Birgitta 是 Thoughtworks 的杰出工程师（Distinguished Engineer）和 AI 辅助交付专家。她拥有超过 20 年的软件开发、架构师和技术领导者经验。

本文是"探索生成式 AI（Exploring Gen AI）"系列的一部分。该系列记录了 Thoughtworks 技术专家探索将生成式 AI（gen ai）技术用于软件开发的过程。

2026 年 2 月 17 日

阅读**OpenAI 最近关于"约束工程（Harness engineering）"的文章**非常有趣，该文章描述了一个团队如何使用"完全没有手动编写的代码"作为强制功能，构建了一个约束环境（harness），用于用 AI Agent 维护一个大型应用程序。5 个月后，他们构建了一个真正的产品，现在代码已超过 100 万行。

这篇文章的标题是"约束工程：在 Agent 优先的世界中利用 Codex"，但正文中只提到"约束（harness）"一次。也许这个术语是受到**Mitchell Hashimoto 最近的博客文章**启发的事后想法。无论如何，我喜欢用"约束（harness）"这个词来描述我们可以用来控制 AI Agent 的工具和实践。

OpenAI 团队的约束组件在 3 个类别中混合了确定性和基于 LLM 的方法（基于我的解释分组）：

- **上下文工程（Context engineering）**：代码库中持续增强的知识库，加上 Agent 对动态上下文（如可观察性数据和浏览器导航）的访问
- **架构约束（Architectural constraints）**：不仅由基于 LLM 的 Agent 监控，还由确定性的自定义 linter 和结构测试监控
- **"垃圾回收（Garbage collection）"**：定期运行的 Agent，用于查找文档中的不一致或违反架构约束的情况，对抗熵增和衰减

他们还强调了这是多么迭代的过程："当 Agent 遇到困难时，我们将其视为一个信号：识别缺少的东西——工具、护栏、文档——并将其反馈回仓库，始终让 Codex 本身编写修复程序。"

所有描述的措施都专注于提高长期内部质量和可维护性。我在文章中缺少的是对功能和行为的验证。

暂且抛开这个空白，假设我们可以相信 OpenAI 对成功的描述（就作者和团队而言，OpenAI 确实有既得利益让我们相信 AI 可维护的代码）——以下是我对文章内容的想法。

### 约束——未来的服务模板？

大多数组织只有两三个主要技术栈——不是每个应用程序都是独特的雪花。这篇文章让我想象这样一个未来：团队从一组常见应用程序拓扑的约束中进行选择，以开始工作。这让我想起了今天的服务模板（service templates），它帮助团队在"黄金路径（golden path）"上实例化新服务。约束——带有自定义 linter、结构测试、基本上下文和知识文档，以及额外的上下文提供者——会成为新的服务模板吗？团队会将它们作为起点，然后随着时间的推移根据应用程序的具体情况进行塑造吗？

使用服务模板，团队随着经验的积累做出贡献，然后其他团队往往难以合并更新。我们会看到约束出现类似的分叉和同步挑战吗？

这篇文章还让我重新审视了一些我早期的假设：

### 运行时必须受到约束才能实现更多的 AI 自主权？

许多早期和当前的 AI 编码炒作都假设 LLM 将为我们提供目标运行时的无限灵活性。用任何语言、任何模式生成，没有约束——LLM 会搞清楚。但对于我们可以信任的大规模可维护 AI 生成代码，必须有所妥协。

所描述的约束表明，增加信任和可靠性需要限制解决方案空间：特定的架构模式、强制边界、标准化结构。这意味着放弃一些"生成任何东西"的提示灵活性，转而使用充满技术细节的规则和约束。

### 收敛到有限数量的技术栈和拓扑？

随着编码变得不那么关于输入代码，而更多关于指导其生成，AI 可能会推动我们走向更少的技术栈。框架和 SDK 的可用性仍然很重要——我们反复看到对人类好的东西对 AI 也好。但在这种细节层面上，开发者的喜好将变得不那么重要。界面中的小效率低下和特性将不那么烦人，因为我们不直接处理它们。我们可能会选择有良好约束可用的技术栈，并优先考虑"AI 友好性"。

这可能不仅适用于技术栈，还适用于代码库结构和拓扑。我们可能会默认使用更容易用 AI 维护的结构，因为它们更容易约束。OpenAI 团队讨论了架构刚性和执行规则。我看到的主要关注领域是保持数据结构稳定以及定义和执行模块边界。听起来很合理——但没有具体例子，我仍然难以想象"我们要求 Codex 在边界解析数据形状"在他们的约束中实际上是什么样子。

但是，如果我们能广泛地弄清楚如何约束代码库设计模式，这些拓扑会成为新的抽象层吗？而不是像许多 AI 爱好者希望的那样是自然语言本身？

### 两个未来世界：AI 前与 AI 后的应用程序维护？

假设我们开发了良好的约束技术，将 AI 自主权提高到 9 级，并增加对结果的信心。哪些技术可以应用于现有应用程序，哪些只适用于从头开始构建并考虑约束的应用程序？

对于较旧的代码库，我们需要考虑 retrofitting 一个约束是否值得付出努力。AI 可以帮助我们更快地做到这一点，但这些应用程序通常如此非标准化且充满熵，可能不值得。这让我想起在一个从未有过静态代码分析工具的代码库上运行它，然后淹没在警报中。

### 你今天有什么约束？

这个团队在他们的约束上工作了 5 个月，这表明这不是为了快速结果而可以仓促投入的事情。但值得反思的是你今天的约束是什么。你有 pre-commit hook 吗？里面有什么？你有关于自定义 linter 的想法吗？你想对代码库施加什么架构约束？你尝试过像 ArchUnit 这样的结构测试框架吗？

### 最后的想法

不出所料，他们描述的内容听起来比只是生成和维护一堆 Markdown 规则文件要做的工作多得多。他们为约束的确定性部分构建了广泛的工具。他们的上下文工程不仅涉及策划知识库，还涉及重要的设计工作——代码设计本身是上下文的重要组成部分。

OpenAI 团队说："我们现在最困难的挑战集中在设计环境、反馈循环和控制系统上。"这让我想起了**Chad Fowler 最近关于"重新定位严谨性（Relocating Rigor）"的文章**。听到关于严谨性可能去向的具体想法和经验令人耳目一新，而不是仅仅希望"更好的模型"会神奇地解决可维护性问题。

最后，这一次，我喜欢这个领域的一个术语。虽然它只有 2 周的历史——我可能会屏住呼吸，直到有人把他们的一次提示、基于 LLM 的代码审查 Agent 称为约束……
