# 将 AI 锚定到参考应用

本文是"探索生成式 AI（Exploring Gen AI）"系列的一部分。该系列记录了 Thoughtworks 技术专家探索将生成式 AI（gen ai）技术用于软件开发的过程。

服务模板（service templates）是组织为其工程团队构建"黄金路径（golden paths）"时的典型构建块，目的是让做正确的事情变得容易。模板应该作为组织中所有服务的榜样，始终代表最新的编码模式和标准。

然而，服务模板的挑战之一是，一旦团队用模板实例化了一个服务，将模板更新反馈给这些服务就很繁琐。生成式 AI（GenAI）能帮忙吗？

## 参考应用作为样本提供者

作为我最近在这里写到的更大实验的一部分，我创建了一个**MCP 服务器**，让编码助手可以访问典型模式的代码样本。在我的案例中，这是针对 Spring Boot Web 应用程序，模式包括仓库（repository）、服务（service）和控制器（controller）类。到目前为止，提供一个经过充分验证的提示实践是：向**LLM**提供我们想要的输出示例会带来更好的结果。用更花哨的术语来说"提供示例"：这也称为"少样本提示（few-shot prompting）"或"上下文学习（in-context learning）"。

当我开始在提示中使用代码样本时，我很快意识到这有多繁琐，因为我在一个自然语言的 Markdown 文件中工作。这感觉有点像在大学里用铅笔写第一次 Java 考试：你不知道你写的代码是否真的能编译。更重要的是，如果你为多个编码模式创建提示，你希望它们彼此保持一致。在可编译和运行的参考应用项目（如服务模板）中维护代码样本，可以更容易地为 AI 提供可编译的、一致的样本。

## 检测与参考应用的漂移

现在回到我在开头提到的问题陈述：一旦代码生成（无论是用 AI 还是用服务模板），然后进一步扩展和维护，代码库往往会偏离参考应用的榜样。

所以在第二步中，我想知道我们如何使用这种方法在代码库和参考应用之间进行"代码模式漂移检测（code pattern drift detection）"。我用一个相对简单的例子测试了这个方法，我在参考应用的控制器类中添加了一个记录器（logger）和 `log.debug` 语句：

然后我扩展了 MCP 服务器以提供对参考应用中 Git 提交的访问。让 Agent 首先查找参考应用中的实际更改，这使我对漂移检测的范围有了一些控制权，我可以使用提交来准确地告诉 AI 我对哪种类型的漂移感兴趣。在我引入这个方法之前，当我只是让 AI 比较参考控制器和现有控制器时，它有点过度了，进行了大量不相关的比较，而我看到这种提交范围限定方法有很好的效果。

在第一步中，我只是让 AI 为我生成一份报告，识别所有漂移，这样我可以审查和编辑该报告，例如删除不相关的发现。在第二步中，我让 AI 接受报告并编写代码来弥补已识别的差距。

### AI 何时带来新的东西？

像添加记录器或更改日志框架这样简单的事情，也可以通过像**OpenRewrite**这样的**codemod**工具确定性地将完成。所以在你选择 AI 之前请记住这一点。

AI 可以发光发热的时候是当我们有漂移需要编码，这比基于正则表达式的 codemod 配方更动态。在日志示例的高级形式中，这可能是将非标准化的、丰富的日志语句转换为结构化格式，其中**LLM**可能更擅长将各种现有日志消息转换为各自的结构。
