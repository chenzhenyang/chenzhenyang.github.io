# 理解规格驱动开发：Kiro、spec-kit 和 Tessl

Birgitta 是 Thoughtworks 的杰出工程师（Distinguished Engineer）和 AI 辅助交付专家。她拥有超过 20 年的软件开发、架构师和技术领导者经验。

本文是"探索生成式 AI（Exploring Gen AI）"系列的一部分。该系列记录了 Thoughtworks 技术专家探索将生成式 AI（gen ai）技术用于软件开发的过程。

2025 年 10 月 15 日

我一直试图理解最新的 AI 编码流行语之一：规格驱动开发（Spec-driven development，SDD）。我研究了三个自称是 SDD 工具的工具，并试图理清它目前意味着什么。

## 定义

就像这个快节奏领域中许多新兴术语一样，"规格驱动开发（SDD）"的定义仍在变化中。以下是我从目前看到的用法中收集到的信息：规格驱动开发意味着在使用 AI 编写代码之前先编写一个"规格（spec）"（"文档优先"）。规格成为人类和 AI 的真实来源（source of truth）。

**GitHub：** "在这个新世界中，维护软件意味着演进规格。[...] 开发的通用语言（lingua franca）提升到了更高的层次，代码是最后一英里的方法。"

**Tessl：** "一种开发方法，其中规格——而不是代码——是主要产物。规格用结构化的、可测试的语言描述意图，Agent 生成代码来匹配它们。"

在查看了该术语的用法以及一些声称实现 SDD 的工具之后，在我看来，实际上存在多个实现层次：

- **规格优先（Spec-first）**：先编写一个经过深思熟虑的规格，然后在手边任务的 AI 辅助开发工作流中使用它。
- **规格锚定（Spec-anchored）**：任务完成后规格仍然保留，继续用于相应功能的演进和维护。
- **规格即源码（Spec-as-source）**：规格随着时间的推移是主要源文件，只有规格由人类编辑，人类从不触碰代码。

我发现的所有 SDD 方法和定义都是规格优先的，但并非所有都致力于成为规格锚定或规格即源码。而且通常对于规格随时间的维护策略应该是什么，往往是模糊的或完全开放的。

## 什么是规格？

就定义而言，关键问题当然是：什么是规格？似乎没有一个通用的定义，我见过的最接近一致定义的是将规格与"产品需求文档（Product Requirements Document）"进行比较。

目前这个术语相当 overloaded，以下是我尝试定义什么是规格：

规格是一个结构化的、面向行为的产物——或一组相关的产物——用自然语言编写，表达软件功能并作为 AI 编码 Agent 的指导。规格驱动开发的每个变体定义了它们对规格的结构、详细程度，以及这些产物如何在项目中组织的方法。

我认为规格和更通用的代码库上下文文档之间有一个有用的区别。通用上下文是诸如规则文件，或产品和代码库的高级描述之类的内容。一些工具称这种上下文为**记忆库（memory bank）**，所以我这里就这样使用。这些文件与代码库中的所有 AI 编码会话都相关，而规格仅与实际上创建或更改该特定功能的任务相关。

## 评估 SDD 工具的挑战

事实证明，以接近实际使用的方式评估 SDD 工具和方法非常耗时。你必须用不同规模的问题、绿地项目（greenfield）、棕地项目（brownfield）来尝试它们，并真正花时间审查和修订中间产物，而不仅仅是粗略浏览。因为正如**GitHub 关于 spec-kit 的博客文章**所说："至关重要的是，你的角色不仅仅是引导。而是验证。在每个阶段，你都要反思和完善。"

对于我尝试的三个工具中的两个，将它们引入现有代码库似乎需要更多的工作，因此更难评估它们对棕地代码库的有用性。在我听到人们在"真实"代码库上使用一段时间的使用报告之前，我仍然有很多关于这在现实生活中如何运作的开放性问题。

话虽如此——让我们来看看这三个工具。我将首先分享它们如何工作的描述（或者更确切地说，我认为它们如何工作），并将我的观察和问题保留到最后。请注意，这些工具发展非常迅速，所以自从我在 9 月使用它们以来，它们可能已经发生了变化。

## Kiro

**Kiro** 是我尝试的三个工具中最简单（或最轻量级）的。它似乎主要是规格优先的，我发现的所有示例都将其用于一个任务或用户故事，没有提到如何在多个任务中随时间以规格锚定的方式使用需求文档。

**工作流：** 需求 → 设计 → 任务

每个工作流步骤由一个 Markdown 文档表示，Kiro 在其基于 VS Code 的分发版中指导你完成这三个工作流步骤。

**需求：** 结构化为需求列表，其中每个需求代表一个"用户故事（User Story）"（采用"As a..."格式），带有验收标准（采用"GIVEN... WHEN... THEN..."格式）

**设计：** 在我的尝试中，设计文档包含下图中看到的部分。我只保留了一次尝试的结果，所以我不确定这是否是一个一致的结构，还是它会根据任务而变化。

**任务：** 一个任务列表，可以追溯到需求编号，并获得一些额外的 UI 元素来逐个运行任务，并审查每个任务的更改。

Kiro 也有记忆库的概念，他们称之为"引导（steering）"。其内容是灵活的，他们的工作流似乎不依赖于任何特定文件的存在（我在发现引导部分之前就进行了使用尝试）。当你让 Kiro 生成引导文档时，它创建的默认拓扑是 `product.md`、`structure.md`、`tech.md`。

## Spec-kit

**Spec-kit** 是 GitHub 的 SDD 版本。它作为 CLI 分发，可以为各种常见的编码助手创建工作区设置。一旦设置了该结构，你就可以通过编码助手中的斜杠命令与 spec-kit 交互。因为它的所有产物都直接放入你的工作区，这是这里讨论的三个工具中可定制性最强的。

**工作流：** 章程 → 𝒢 规格 → 计划 → 任务 𝒢

Spec-kit 的记忆库概念是规格驱动方法的前提条件。他们称之为**章程（constitution）**。章程应该包含"不可变的"高级原则，应该始终应用于每个更改。它基本上是一个非常强大的规则文件，被工作流大量使用。

在每个工作流步骤（规格、计划、任务）中，spec-kit 借助 bash 脚本和一些模板实例化一组文件和提示。然后工作流大量使用文件中的检查清单，来跟踪必要的用户澄清、章程违规、研究任务等。它们就像每个工作流步骤的"完成定义（definition of done）"（尽管由 AI 解释，所以没有 100% 的保证它们会被遵守）。

乍一看，GitHub 似乎**致力于规格锚定方法**（"这就是为什么我们正在重新思考规格——不是作为静态文档，而是作为与项目一起演进的、活的、可执行的产物。规格成为共享的真实来源。当某些事情没有意义时，你回到规格；当项目变得复杂时，你完善它；当任务感觉太大时，你分解它们。"）然而，spec-kit 为创建的每个规格创建一个分支，这似乎表明他们将规格视为变更请求生命周期内的活产物，而不是功能的生命周期。**这个社区讨论**正在谈论这种困惑。这让我觉得 spec-kit 仍然只是我所说的规格优先，而不是随时间规格锚定。

## Tessl Framework

（仍在私人测试阶段）

与 spec-kit 一样，**Tessl Framework** 作为 CLI 分发，可以为各种编码助手创建所有工作区和配置结构。CLI 命令还兼作 MCP 服务器。

Tessl 是这三个工具中唯一一个明确致力于规格锚定方法，甚至正在探索 SDD 的规格即源码层次的工具。Tessl 规格可以作为正在维护和编辑的主要产物，代码甚至在顶部用注释标记为 `// GENERATED FROM SPEC - DO NOT EDIT`。目前这是规格和代码文件之间的 1:1 映射，即一个规格转换为代码库中的一个文件。但 Tessl 仍处于测试阶段，他们正在尝试不同版本的方法，所以我可以想象这种方法也可以在一个规格映射到具有多个文件的代码组件的层次上采用。alpha 产品将支持什么还有待观察。（Tessl 团队自己将他们的框架视为比他们当前的公共产品 Tessl Registry 更未来的东西。）

```
// GENERATED FROM SPEC - DO NOT EDIT
```

以下是我让 Tessl CLI 从现有代码库中的 JavaScript 文件逆向工程（`tessl document --code ...js`）的规格示例：

```
tessl document --code ...js
```

像 `@generate` 或 `@tests` 这样的标签似乎告诉 Tessl 要生成什么。API 部分展示了在规格中定义至少暴露给代码库其他部分的接口的想法，大概是为了确保生成的组件的这些更关键部分完全在维护者的控制之下。为此规格运行 `tessl build` 会生成相应的 JavaScript 代码文件。

```
@generate
```

```
@test
```

```
tessl build
```

将规格即源码的规格放在相当低的抽象层次，每个代码文件，可能会减少 LLM 必须执行的步骤和解释的数量，从而减少错误的机会。即使在这个低抽象层次上，当我从同一规格多次生成代码时，我也看到了非确定性的实际作用。迭代规格并使其越来越具体以提高代码生成的可重复性是一个有趣的练习。这个过程让我想起了编写无歧义和完整规格的一些陷阱和挑战。

## 观察和问题

这三个工具都将自己标记为规格驱动开发的实现，但它们彼此之间相当不同。所以在谈论 SDD 时要记住的第一件事是，它不只是一件事。

### 一个工作流适合所有规模？

Kiro 和 spec-kit 各提供一个有意见的工作流，但我相当确定它们都不适合大多数现实生活中的编码问题。特别是，我不太清楚它们将如何适应足够不同的问题规模以普遍适用。

当我让 Kiro 修复一个小 bug 时（这是我过去用来尝试 Codex 的同一个 bug），很快就清楚工作流就像用大锤砸坚果。需求文档将这个小 bug 变成了 4 个"用户故事"，总共有 16 个验收标准，包括这样的宝石："用户故事：作为一名开发者，我希望转换函数能够优雅地处理边缘情况，以便在引入新类别格式时系统保持稳健。"

我使用 spec-kit 时也有类似的挑战，我不太确定应该用它来处理什么规模的问题。可用的教程通常基于从头创建应用程序，因为这对教程来说最容易。我最终尝试的用例之一是一个功能，在我过去的团队中这将是一个 3-5 点的故事。该功能依赖于很多已经存在的代码，它应该构建一个概述模态框，总结现有仪表板中的一堆数据。随着 spec-kit 采取的步骤数量，以及它为我创建的要审查的 Markdown 文件数量，这再次感觉对于问题的规模来说是杀鸡用牛刀。这是一个比我用 Kiro 更大的问题，但也是一个更复杂的工作流。我甚至没有完成完整的实现，但我认为在我运行和审查 spec-kit 结果所花的相同时间内，我本可以用"普通"AI 辅助编码实现该功能，而且我会感觉更有控制力。

一个有效的 SDD 工具至少必须为几种不同的核心工作流提供灵活性，以适应不同规模和类型的更改。

### 审查 Markdown 胜过审查代码？

正如刚才提到的，正如你在上面的工具描述中看到的那样，spec-kit 创建了大量的 Markdown 文件供我审查。它们是重复的，彼此之间以及与已经存在的代码之间。有些已经包含了代码。总的来说，它们非常冗长，审查起来很繁琐。在 Kiro 中稍微容易一些，因为你只得到 3 个文件，而且"需求 > 设计 > 任务"的心智模型更直观。然而，如前所述，Kiro 对于我要求它修复的小 bug 来说也太冗长了。

老实说，我宁愿审查代码也不愿审查所有这些 Markdown 文件。一个有效的 SDD 工具必须提供非常好的规格审查体验。

### 虚假的控制感？

即使有所有这些文件、模板、提示、工作流和检查清单，我也经常看到 Agent 最终没有遵循所有指令。是的，上下文窗口现在更大了，这经常被提及为规格驱动开发的推动因素之一。但仅仅因为窗口更大，并不意味着 AI 会正确地拾取其中的所有内容。

例如：Spec-kit 在计划期间的某个地方有一个研究步骤，它对现有代码和已经存在的内容做了很多研究，这很棒，因为我要求它添加一个基于现有代码的功能。但最终 Agent 忽略了这些是现有类描述的注释，只是将它们作为新规格并重新生成它们，创建了重复。但我不仅看到了忽略指令的示例，我还看到 Agent 因为过于急切地遵循指令而做得过火（例如章程条款之一）。

过去表明，我们保持对所构建内容的控制的最佳方式是小步迭代，所以我非常怀疑大量前期规格设计是否是一个好主意，尤其是当它过于冗长时。一个有效的 SDD 工具必须适应迭代方法，但小工作包几乎似乎与 SDD 的理念相悖。

### 如何有效地将功能规格与技术规格分离？

在 SDD 中，有意分离功能规格和技术实现是一个共同的理念。我猜潜在的希望是，最终我们可以让 AI 填写所有解决方案和细节，并用同一规格切换到不同的技术栈。

在现实中，当我尝试 spec-kit 时，我经常困惑何时应该保持在功能层次，何时是添加技术细节的时候。教程和文档在这方面也不完全一致，似乎对"纯功能"真正意味着什么有不同的解释。当我回想起在我的职业生涯中读过的许多、许多没有正确分离需求和实现的用户故事时，我认为我们作为一个行业在这方面没有良好的记录。

### 目标用户是谁？

许多规格开发工具的演示和教程包括定义产品和功能目标之类的内容，它们甚至结合了"用户故事"这样的术语。这里的想法可能是使用 AI 作为跨技能培养的推动者，让开发者更多地参与需求分析？或者让开发者在产品人员处理这个工作流时与他们配对？但这些都没有明确说明，而是作为给定的前提，即开发者会完成所有这些分析。

在这种情况下，我会再次问自己，SDD 旨在解决什么规模和问题类型？可能不是针对仍然非常不清楚的大型功能，因为这肯定需要更专业的产品和需求技能，以及许多其他步骤，如研究和利益相关者参与？

### 规格锚定和规格即源码：我们是否从过去学习？

虽然许多人将 SDD 与 TDD 或 BDD 进行类比，但我认为对于规格即源码来说，另一个重要的并行是查看 MDD（模型驱动开发，model-driven development）。我在职业生涯初期参与过一些大量使用 MDD 的项目，当我尝试 Tessl Framework 时，我一直在想起这一点。MDD 中的模型基本上是规格，虽然不是用自然语言，而是用例如自定义 UML 或文本 DSL 表达。我们构建了自定义代码生成器来将这些规格转换为代码。

最终，MDD 从未在业务应用程序中起飞，它处于尴尬的抽象层次，只是产生了太多的开销和约束。但 LLM 消除了 MDD 的一些开销和约束，所以有了新的希望，我们现在终于可以专注于编写规格并从中生成代码。使用 LLM，我们不再受预定义的、可解析的规格语言的约束，也不必构建复杂的代码生成器。当然，代价是 LLM 的非确定性。而且可解析的结构也有我们现在正在失去的优点：我们可以为规格作者提供大量工具支持来编写有效、完整和一致的规格。我想知道规格即源码，甚至规格锚定，是否最终会同时具有 MDD 和 LLM 的缺点：**不灵活性**和**非确定性**。

要明确的是，我并不是对我过去的 MDD 经历感到怀旧，说"我们不妨把它带回来"。但当我们今天探索规格驱动时，我们应该回顾过去的从规格生成代码的尝试，从中学习。

## 结论

在我个人使用 AI 辅助编码时，我也经常花时间先精心制作某种形式的规格给编码 Agent。所以规格优先的一般原则在许多情况下绝对是有价值的，关于如何构建该规格的不同方法是非常受欢迎的。它们是我目前从从业者那里听到的最经常被问到的问题之一："我如何构建我的记忆库？"，"我如何为 AI 编写好的规格和设计文档？"。

但"规格驱动开发"这个术语还没有很好的定义，它已经**语义扩散**了。我最近甚至听到人们基本上将"规格"用作"详细提示"的同义词。

关于我尝试过的工具，我在这里列出了我对它们实际世界有用性的许多疑问。我想知道它们中的一些是否过于字面地将 AI Agent 与我们的现有工作流一起输入，最终放大了现有的挑战，如审查过载和幻觉。特别是对于创建大量文件的更复杂的方法，我不禁想起德语复合词"Verschlimmbesserung"：我们是否在试图改进某事时使其变得更糟？
